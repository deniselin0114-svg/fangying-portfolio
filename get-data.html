Fabric Real-Time Intelligence: Get data

Transforming fragmented, inconsistent ingestion flows into a unified, intuitive, end-to-end experience that helps users bring real-time data into Fabric quickly and confidently.

⤓
Overview

A unified ingestion experience for Real-Time Intelligence

  • Role — solo Product Designer owning the end-to-end UX, partnering with 4+ product managers across different aspects of the Get data flow and 2 development teams (20+ engineers).
  • Scope — the RTI ingestion experience across 3+ teams and time zones (Eventstream, Real-Time hub, Eventhouse, and more).
  • Impact — a universal experience to ingest data across multiple items, improving usability, reducing drop-offs, and enabling complete ingestion workflows.
Problem statement

A fragmented and confusing ingestion experience

From research and internal discussions, we identified several critical issues:

  • Multiple disconnected “Get data” experiences across RTI workloads.
  • Users had to switch between different wizards and mental models.
  • The lack of an end-to-end flow (source → destination) caused incomplete setups.

Users were dropping off mid-flow

  • Many users successfully connected their data…
  • …but never completed the pipeline by adding a destination.
Result

Data was ingested into Fabric but never used or activated — leading to a poor perception of product value.

Design goal

A unified, intelligent ingestion experience

An experience that:

  • Works across all RTI entry points.
  • Guides users from source selection → ingestion → destination.
  • Reduces cognitive load while still supporting advanced scenarios.
  • Delivers a consistent experience across workloads.
  • Lets users ingest data quickly and reliably with minimal configuration.
Key design principles & solution

Browse, connect, and complete — end to end

We divided the flow into three aspects: the browsing state, the connecting steps, and the end-to-end behavior.

Browse

For each starting point — the Real-Time hub, the Eventstream streaming pipeline item, and destinations like Eventhouse and Activator — the browsing experience keeps the same look and feel. Drawing on the strong completion and success rates we measured from the Get data wizard across Microsoft products, we inherited the wizard experience already supported in Microsoft 365, Power BI, and Fabric, and extended it into a streaming data browsing experience.

We segment users into day-0 users and users without streaming data, and give each the right browsing experience. If a user is a Microsoft customer who already has Azure data or Dataverse data, we make that data quick and easy to access. For users with no existing data — day-0 and new-to-Microsoft — we recommend the right data in the very first browsing experience.

Together with the research team, we ran multiple user studies on the Get data experience and validation. One key finding:

Start with data. In the browsing state, always start with the data source. Instead of asking users to choose by technology (real-time streaming, batch copy, etc.), we found that while browsing, users care about which source they're looking for — they consistently have a specific data source in mind.

Connecting steps

After mapping how users move from different entry points, I designed a universal set of connection steps validated against users' mental models. After selecting or browsing a data source, the steps are:

  • Configure the data source — create connections, select a topic, consumer group, choose a dataset, and so on.
  • Preview the data stream — including preparing the data (deciding schemas) and updating the item that manages the streaming pipeline.
  • Configure the destination — an optional step where users decide where to route the data.
  • Finalize and run.

System guided, user controlled. We recommend default options automatically while leaving flexibility for advanced users. Based on the data source, entry point, and more, we default the way users ingest their streaming data.

  • From telemetry on Eventhouse users, we know they are far more likely to ingest data from Azure Event Hubs directly without transforming it (82%+). For these cases, we customize the experience to default to creating a shortcut to the Azure Event Hub, rather than copying the data into Fabric.
  • Based on the type of source (CDC, streaming data, discrete events, etc.) we also default the technical behavior. For CDC sources, we know the schema is fixed — so we default to saving the schema to a universal schema-management item and automatically applying schema changes to data validation and destination table mapping.

Progressive complexity. Keep the default experience simple and reveal advanced options only when needed. Given the complexity of data ingestion, we provide strong defaults grounded in telemetry, research, and user studies, and surface only the necessary setup in the UI — reducing drop-off and distraction.

End-to-end behavior

Once users finish the wizard, the end-to-end flow tells the right story and gives the right suggestions for next steps, so users understand where their data goes and what to do next.

Maintain context. Respect the entry point so users understand where they came from and where they are, with clear options to jump to other items and why. The wizard-style ingestion also adapts the experience dynamically to that context.

Decisions
  1. Lead the browse experience with the data source, not the ingestion technology.
  2. Default the setup to avoid confusion and complexity.
  3. Use a data-driven approach to suggest different default options.
  4. Maintain users in their context.
Key features

What the unified experience delivers

1. Unified entry across all RTI workloads

  • The same experience from Eventstream, Real-Time hub, Eventhouse, and more.
  • Eliminates fragmentation.
Solves

Inconsistent wizards and the cognitive cost of switching between mental models.

2. Built-in data preview, early in the flow

  • Users can preview data before ingestion.
  • Supports column selection, filtering, and schema understanding.
Solves

The gap in the current system, where the lack of data preview makes validation difficult.

3. Integrated destination step (end-to-end flow)

  • Destination configuration is part of the wizard, not a separate step.
  • Prevents incomplete ingestion setups.
Solves

User drop-off after data connection and unclear next steps.

4. Smart defaults & system recommendations

  • Automatically suggests an ingestion method and compatible destinations.
  • Users can override when needed.

5. Support for advanced scenarios

  • Schema editing
  • Table mapping
  • Delta flow (CDC)
  • Event Hub mounting
Design challenges & tradeoffs

Balancing consistency, power, and context

1. Handling different backend behaviors

Eventstream and KQL have different preview logic:

  • Eventstream — preview from the source.
  • KQL — sample the actual ingested data.
Challenge

Create a consistent UX despite different backend implementations.

2. Balancing simplicity vs. power

  • Beginners need guidance.
  • Experts need flexibility.
Solution

Progressive disclosure — keep the “happy path” simple.

3. Supporting multiple entry points

  • Users start from different contexts.
  • Each context carries different expectations.
Solution

Context-aware defaults and adaptive destination selection.

Designer reflection

What I learned

  • End-to-end thinking matters. Designing only “getting data” wasn't enough — users needed complete workflows.
  • Visibility drives confidence. Users won't proceed without seeing the data, so preview became a core design pillar.
  • Systems design > single-feature design. This project required aligning multiple teams (Eventstream, KQL, RTI), multiple workflows, and multiple user mental models.
  • Simplification is strategic. Instead of adding features, we focused on reducing decisions and guiding users.
